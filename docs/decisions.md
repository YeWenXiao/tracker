# 版本决策记录

## 决策 #1: v1.5 和 v2.0 并行开发 (2026-03-17)

**背景:** v1.0 有两个独立的痛点需要解决
- 网络视频流延迟高 → v2.0 用 MIPI 解决
- 换目标需要重启 → v1.5 用热加载解决

**决策:** 两个版本在独立分支并行开发，互不影响

**后续计划:** 两个分支的改进都稳定后，合并到一个新版本（v2.5 或 v3.0）

---

## 决策 #2: Round 2 优化方向 (2026-03-17)

**Round 1 审查发现的问题和优化点：**

### v2.0-mipi 需要优化：
1. **CUDA 加速预处理** - cvtColor/resize 用 cv2.cuda 模块，减少 CPU 开销
2. **配置文件** - 硬编码的 GStreamer 参数应提取为 config.yaml
3. **自动 fallback** - MIPI 打开失败时自动降级到 RTSP，无需手动切参数
4. **帧率限制逻辑** - --fps-cap 参数应该在取帧端限流，不是靠 GStreamer

### v1.5-hotswap 需要优化：
1. **增量加载** - 当前 reload 是全量重算特征，应检测新增/删除的模板只做增量更新
2. **注释改回中文** - 原代码是中文注释，v1.5 改成了英文，不统一
3. **HTTP API 集成到 recognize.py** - 当前 target_server.py 是独立文件，应提供 --api 参数在识别引擎内启动
4. **WebSocket 通知** - 换目标后推送通知给前端

## 决策 #3: Round 3 优化方向 (2026-03-17)

### v2.0-mipi Round 3:
1. **CUDA 加速模板匹配** - cv2.cuda.matchTemplate 替代 CPU 版本，快速模式预计从29ms降到<10ms
2. **性能 benchmark 脚本** - 自动对比 MIPI vs RTSP 各项指标
3. **帧同步优化** - 识别线程和采集线程解耦，避免识别慢时丢帧

### v1.5-hotswap Round 3:
1. **目标置信度权重系统** - 不同目标可设不同阈值
2. **目标模板预览 API** - GET /api/targets/<name>/preview 返回缩略图
3. **WebSocket 实时通知** - 换目标/识别结果实时推送

## 决策 #4: Round 4 - 稳定性+实用功能 (2026-03-17)

### v2.0-mipi Round 4:
1. **日志系统** - logging 替代 print，支持文件输出和远程查看
2. **ROI 区域识别** - 用户可框选感兴趣区域，只在 ROI 内做匹配（减少计算量）
3. **运行状态监控** - 在 OSD 上显示 CPU/GPU 温度、内存使用、识别队列深度

### v1.5-hotswap Round 4:
1. **目标模板版本管理** - 每次换目标记录历史，支持回滚到之前的目标集
2. **批量导入/导出** - ZIP 打包上传/下载整套目标模板
3. **识别结果历史** - 最近N次识别结果缓存，API可查询

## 决策 #5: Round 5 - 核心能力提升 (2026-03-17)

**方向:** 两个版本都已足够成熟，Round 5 聚焦于核心识别/追踪能力提升。

### v2.0-mipi Round 5:
1. **卡尔曼滤波追踪** - 识别到目标后启动追踪，预测下一帧位置，减少全帧搜索
2. **自适应识别频率** - 追踪稳定时降低识别频率（每3帧做一次），提升整体帧率
3. **云台自动跟踪** - 识别到目标后通过 siyi_sdk 控制云台居中目标

### v1.5-hotswap Round 5:
1. **目标相似度去重** - 上传新目标时检测与已有目标的相似度，防止重复
2. **识别置信度自适应** - 根据历史统计自动调整阈值
3. **多客户端状态同步** - 多个浏览器打开管理页面时状态同步

## 决策 #6: Round 6 - 代码质量+鲁棒性 (2026-03-17)

两个版本功能已非常丰富，Round 6 聚焦代码质量、错误处理和边缘场景。

1. **异常恢复** - 视频流断开自动重连、CUDA OOM 回退CPU、云台通信超时处理

3. **管理页面国际化** - 中文界面

## 决策 #8: Round 8 - 性能极限+部署 (2026-03-18)
